HTML
The skeleton
HyperText Markup Language. Defines the structure and content of the page. Every heading, paragraph, image, link, and section is an HTML element.
index.html

  <nav>...</nav>
</header>

How a browser turns
code into a page

You need a mental model of what Copilot is producing.

1

Copilot writes code

HTML, CSS, and JS files are generated based on your prompt. Each is a plain text file.

2

You save the files

index.html, style.css, script.js — all in one folder. The HTML file links to the CSS and JS files.

3

GitHub Pages hosts the files

You push the folder to GitHub. GitHub Pages makes the files available at a public URL.

4

The browser requests the files

When someone visits your URL, the browser downloads the files and reads them in order.

5

The browser renders the page

HTML provides structure, CSS applies styling, JS adds behaviour. The page appears on screen.

Your job is step 1. Copilot does the writing. GitHub does steps 3–5 automatically.

What Copilot is doing
when it generates code

Copilot is a large language model. It predicts code the same way it predicts text — based on your prompt and its training data.

01

It reads your prompt as instructions

The more specific your prompt, the more specific the output. Vague prompts produce generic code. Precise prompts — with exact colours, fonts, and class names — produce targeted code.

02

It does not know your Figma file

Copilot has never seen your design. Every visual decision — colour HEX, font name, spacing value — must be written into your prompt. It cannot infer from a screenshot.

03

It makes mistakes

Copilot will produce code that does not match your mockup. It will use wrong colours, wrong fonts, wrong spacing. This is expected. Evaluation and iteration are part of the process.

04

It cannot test its own output

Copilot does not open a browser. It does not know if its code actually looks right. You must copy the code, open it in a browser, and compare it to your mockup. That comparison is your job.

05

Context accumulates in a conversation

In one Copilot conversation, earlier prompts inform later ones. Keep your Layer 1, Layer 2, and Layer 3 sessions in separate conversations — or provide context explicitly at the start of each.

The Layered
Prompting Method

Why we build in layers — not all at once.

✗ Build everything at once
✗One massive prompt → one massive output
✗Hard to evaluate — too many things at once
✗When something goes wrong, you cannot isolate the problem
✗Copilot gets confused by complex multi-layer instructions
✗If you need to change the colour, the whole thing needs rewriting
✓ Build in four layers
✓One focused prompt per layer → manageable output
✓Easy to evaluate — one concern at a time
✓Errors are isolated to the layer where they occur
✓Copilot stays focused on a single job at a time
✓Change a colour? Layer 2 prompt only — structure stays intact

Rule: Each layer builds on the one before. Never skip a layer.

Structure

A correctly structured HTML file. No styles. No colour. Just the skeleton — and that is exactly the goal.

What Layer 1 produces

<!DOCTYPE html>
<html lang="en">
<head>
  <title>...</title>
</head>
<body>
  <header>
    <nav>...</nav>
  </header>
  <main>
    <section class="hero">
      <h1>...</h1>
      <p>...</p>
      <a href="#">...</a>
    </section>
  </main>
  <footer>...</footer>
</body>
</html>

✓ Plain is the goal. This is correct.

Semantic HTML — the elements you need

Semantic elements describe their purpose. They help browsers, screen readers, and search engines understand your page.

<header>Top section — contains your logo and navigation. One per page.
<nav>Navigation links. Always inside <header>.
<main>The primary content area. Everything between header and footer. One per page.
<section>A thematic grouping of content. Needs a heading. Use class names.
<article>A self-contained piece of content — a card, a post, a tile. Can be repeated.
<footer>Bottom section — copyright, secondary links, contact. One per page.
<h1>–<h6>Headings in order of importance. Only one <h1> per page. Never skip levels.
<img>Images. Always include an alt attribute. alt="" is required.
The Layer 1 Prompt Template

Copy this structure. Fill in the square brackets. Do not deviate from the layer constraint — the explicit "no CSS" instruction is critical.

Layer 1 Prompt Template — Copilot 
I am building a [category] website called [business name].
Please write a complete HTML skeleton for my homepage
using semantic HTML5 only.

Do not add any CSS.
Do not add any inline styles.
Structure only.

The page must include these sections in this order:

1. <header> containing a <nav> with 5 links:
   [Home] [About] [Menu] [Gallery] [Contact]

2. <main> containing:
   — <section class="hero"> with an <h1>, a <p>, and an <a> CTA button
   — <section class="features"> with three <article> cards,
     each containing <h2>, <p>
   — <section class="gallery"> with six <figure> elements

3. <footer> with copyright text, secondary nav links,
   and social media placeholders

Add a class to every section.
Use meaningful class names matching the section content.
Add a placeholder alt attribute to any <img> elements.
From Copilot's chat to a real file

Copilot's response appears as text in a chat window — it is not automatically a file on your computer. Here's how to turn it into one:

1

Copy the code block

In Copilot's response, find the code block (it will be in a box, usually with a "Copy" button in the corner). Click Copy, or select all the text inside the box and copy it manually.

2

Open a plain text editor

Notepad (Windows), TextEdit (Mac, switch to Plain Text mode via Format menu), or VS Code if you have it.

3

Paste and save as index.html

Save the file with the exact name index.html — not index.txt or index.html.txt. On Mac TextEdit, use "Save As" and manually type the .html extension; on Windows Notepad, change "Save as type" to "All Files" so it does not add .txt automatically.

4

Upload it

This is the file you'll upload to GitHub in the Setup Lab's Step 3.

You'll repeat this same copy → paste → save process for style.css and script.js in later weeks — same idea, different filename and extension.

Evaluating
Your Layer 1 Output

Before you move to Layer 2, verify your skeleton against this checklist.

✗ Reject if you see this
✗Any style= attribute in the HTML
✗Any <style> block in the <head>
✗Any colour values — #XXXXXX anywhere
✗Any font-family references
✗Divs used instead of semantic elements
✗Sections without class names
✗Missing alt attributes on images
✓ Accept when you see this
✓<header>, <nav>, <main>, <section>, <footer> all present
✓Sections have class names matching their content
✓One <h1> per page, <h2> inside sections
✓All links use <a href="#"> placeholder
✓Images have alt="" attributes
✓No inline styles anywhere
✓Opens in browser as plain, readable, unstyled HTML

1

Create a GitHub account

github.com → Sign up. Use your Bond email. Choose a professional username — this becomes part of your URL.

2

Create a new repository

New repository → name it exactly yourusername.github.io (replace yourusername with your actual GitHub username) → Public → Add README → Create repository. This exact naming is what makes GitHub Pages publish your site at the root of your username, not at a sub-path — this is the same repo you already created in the Week 9 lab, not a second one.

3

Upload your files

Add file → Upload files. Upload your index.html first. Then style.css and script.js as they are created each week.

4

Enable GitHub Pages

Settings → Pages → Source: main branch → folder: / (root) → Save. Your URL appears within 60 seconds.

5

Visit your live URL

Copy the URL. Open it in a new tab. Your HTML skeleton is now live on the internet. Share it with no one until it looks right.
