HTML · CSS · JAVASCRIPT

Three languages, one website

You do not need to write any of these

You need to know what they do so you can direct Copilot and evaluate its output.

HTML

The skeleton

+

CSS

The skin

+

JS

The brain

Together they make a working page; structure, style, and behaviour.

HTML: The skeleton

HyperText Markup Language. Defines the structure and content of the page. Every heading, paragraph, image, link, and section is an HTML element.

Layer 1 is 100% HTML. No colour. No fonts. Just structure.

<header>
  <nav>...</nav>
</header>
<main>
  <section class="hero">
    <h1>Welcome</h1>
  </section>
</main>

CSS: The skin

Cascading Style Sheets. Controls how the HTML looks. Colour, typography, spacing, layout, and responsiveness are all CSS.

Layers 2, 3, and 4 are CSS. Your design system tokens become CSS values.

header {
  background: #0F172A;
  padding: 16px 32px;
}
h1 {
  font-size: 48px;
  color: #3B82F6;
}

JS: The brain

JavaScript. Adds interactivity; hamburger menus opening, carousels moving, forms validating. For this course, Copilot handles all JS.

You will need JS for your hamburger nav. Copilot writes it. You evaluate whether it works.

// Hamburger menu toggle
button.addEventListener('click',
  () => nav.classList
    .toggle('open')
);
The pipeline

How a browser turns code into a page

You do not need to understand this deeply. You need a mental model of what Copilot is producing.

1

Copilot writes code

HTML, CSS, and JS files are generated based on your prompt. Each is a plain text file.

2

You save the files

index.html, style.css, script.js, all in one folder. The HTML file links to the CSS and JS files.

3

GitHub Pages hosts the files

You push the folder to GitHub. GitHub Pages makes the files available at a public URL.

4

The browser requests the files

When someone visits your URL, the browser downloads the files and reads them in order.

5

The browser renders the page

HTML provides structure, CSS applies styling, JS adds behaviour. The page appears on screen.

Your job is step 1. Copilot does the writing. GitHub does steps 3–5 automatically.

03

MICROSOFT COPILOT

Know your tool

What Copilot is doing when it generates code

Copilot is a large language model. It predicts code the same way it predicts text, based on your prompt and its training data.

It reads your prompt as instructions

The more specific your prompt, the more specific the output. Vague prompts produce generic code. Precise prompts — with exact colours, fonts, and class names — produce targeted code.

It does not know your Figma file

Copilot has never seen your design. Every visual decision; colour HEX, font name, spacing value — must be written into your prompt. It cannot infer from a screenshot.

It makes mistakes

Copilot will produce code that does not match your mockup. It will use wrong colours, wrong fonts, wrong spacing. This is expected. Evaluation and iteration are part of the process.

It cannot test its own output

Copilot does not open a browser. It does not know if its code actually looks right. You must copy the code, open it in a browser, and compare it to your mockup. That comparison is your job.

Context accumulates in a conversation

In one Copilot conversation, earlier prompts inform later ones. Keep your Layer 1 session, your Layer 2 session, and so on in separate conversation or provide context explicitly.

04

THE LAYERED PROMPTING METHOD

Why build in layers

Why we build in layers, not all at once

Build everything at once

✗One massive prompt → one massive output
✗Hard to evaluate, too many things at once
✗When something goes wrong, you cannot isolate the problem
✗Copilot gets confused by complex multi-layer instructions
✗If you need to change the colour, the whole thing needs rewriting

Build in four layers

✓One focused prompt per layer → manageable output
✓Easy to evaluate, one concern at a time
✓Errors are isolated to the layer where they occur
✓Copilot stays focused on a single job at a time
✓Change a colour? Layer 2 prompt only, structure stays intact
The build order

The four layers

Each layer builds on the one before. Never skip a layer.

Layer 1

Structure (HTML only)

Semantic HTML skeleton. No CSS. Every section, heading, nav, main, footer, and article element in the correct nesting order. No colour. No fonts.

→ Today: Week 9 lab
Layer 2

Styling (CSS colour, type, spacing)

Apply your design system. Background colours from your Figma palette. Google Fonts linked. 8px spacing scale applied. No responsiveness yet.

→ Week 10
Layer 3

Responsive (Breakpoints + mobile layout)

Media queries for mobile (375px) and tablet (768px). Hamburger nav. Grid collapse to single column. Font size adjustments.

→ Week 11
Layer 4

Refinement (Fix, match, polish)

Specific fixes comparing your Figma mockup to your live site. Target individual elements that are wrong. The comparison method in full.

→ Week 11

See what Copilot actually built

Inspecting your page with Chrome DevTools

Reading code is not the same as seeing it render. DevTools lets you click any element on the page and see exactly which HTML produced it — and later, which CSS is styling it.

1

Open DevTools

Right-click anywhere on the page → Inspect. Or press F12 / Ctrl+Shift+I (Windows) / Cmd+Option+I (Mac).

2

Find the Elements panel

It opens by default. This is a live, navigable tree of every tag on the page; your full HTML skeleton, exactly as the browser parsed it.

3

Hover to highlight

Hover any line in the Elements tree and the matching region lights up on the page itself. This is the fastest way to answer "which tag is this?"

4

Click to select, then read Styles

Click an element and the Styles pane on the right lists every CSS rule touching it, including styles your skeleton shouldn't have yet.

5

Edit live to test, don't save here

Click any value in the Styles pane and type a new one. The page updates instantly. This is sandboxed: refreshing the page resets everything. Use it to test ideas, then write the real fix in your Copilot prompt.

Reference

Semantic HTMl; the elements you need

Semantic elements describe their purpose. They help browsers, screen readers, and search engines understand your page.

<html>
<head>
<body>
<header>
<nav>
<main>
<section>
<article>
<footer>

Everything nests inside <html>. <head> and <body> are siblings; everything visible on the page lives inside <body>.

<header>

Top section — contains your logo and navigation. One per page.

<nav>

Navigation links. Goes inside <header>.

<main>

The primary content area. Everything between header and footer. One per page only.

<section>

A thematic grouping of content — hero, features, cards. Needs a heading.

<article>

A self-contained piece of content — a card, a blog post, a product tile. Can be repeated.

<footer>

Bottom section — copyright, secondary links, contact. One per page.

<h1>–<h6>

Headings in order of importance. Only one <h1> per page. Never skip levels.

<p>

Paragraphs of body text.

<a href="">

Links. Used for navigation items and CTAs. href is required.

<img>

Images. Always include an alt attribute. alt="" is required.

Use this today

The Layer 1 prompt template

Copy this structure. Fill in the square brackets with your specifics. Do not deviate from the layer constraint.

prompt template
I am building a [category] website called [business name]. Please write a complete HTML skeleton for my homepage using semantic HTML5 only. Do not add any CSS. Do not add any inline styles. Structure only.

The page must include these sections in this order:
1. <header> containing a <nav> with 5 links: [Home] [About] [Menu] [Gallery] [Contact]
2. <main> containing:
   — <section class="hero"> with an <h1>, a <p>, and an <a> CTA button
   — <section class="features"> with three <article> cards, each containing <h2>, <p>
   — <section class="gallery"> with six <figure> elements
3. <footer> with copyright text, secondary nav links, and social media placeholders

Add a class to every section. Use meaningful class names matching the section content.
Add a placeholder alt attribute to any <img> elements.

→ State your category and business name upfront — gives Copilot context

→ The explicit 'no CSS' constraint is critical — without it, Copilot adds styles

Your lab sheet has the full template with all four section variations. Use it today.

Evaluate before you move on

Evaluating your Layer 1 output

Before you move to Layer 2, verify your skeleton against this checklist.

Reject if you see this

✗Any style= attribute in the HTML
✗Any <style> block in the <head>
✗Any colour values — #XXXXXX anywhere
✗Any font-family references
✗Divs used instead of semantic elements (header, main, footer)
✗Sections without class names
✗Missing alt attributes on images

Accept when you see this

✓<header>, <nav>, <main>, <section>, <footer> all present
✓Sections have class names matching their content
✓One <h1> per page, <h2> inside sections
✓All links use <a href="#"> placeholder
✓Images have alt="" attributes
✓No inline styles anywhere
✓Opens in browser as plain, readable, unstyled HTML

GITHUB PAGES

Your live address

GitHub Pages — your live website address

GitHub Pages turns a GitHub repository into a live website. Free. No server setup. Your site is live in under two minutes.

https://yourusername.github.io/your-repo-name
← This is what you will submit for Assessment 2
1

Create a GitHub account

github.com → Sign up. Use your Bond email. Choose a professional username — this becomes part of your URL.

2

Create a new repository

New repository → name it (e.g. digm11110-portfolio) → Public → Add README → Create repository.

3

Upload your files

Add file → Upload files. Upload your index.html first. Then style.css and script.js as they are created.

4

Enable GitHub Pages

Settings → Pages → Source: main branch → folder: / (root) → Save. Your URL appears within 60 seconds.

5

Visit your live URL

Copy the URL. Open it in a new tab. Your HTML skeleton is now live on the internet. Share it with no one until it looks right.
